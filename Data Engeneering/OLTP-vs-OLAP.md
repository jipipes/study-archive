# OLTP vs. OLAP 비교 분석

데이터 엔지니어링의 핵심은 용도에 맞는 데이터 시스템을 설계하는 것이다.  
비즈니스 운영을 위한 OLTP와 분석을 위한 OLAP의 차이를 정리한다.

---

## 1. OLTP (Online Transaction Processing)
> "현재의 비즈니스를 운영하는 시스템"

일상적인 거래(Transaction)를 실시간으로 처리하며, 주로 서비스의 백엔드 데이터베이스로 사용된다.

- **목적:** 실시간 데이터 처리, 무결성 유지, 빠른 응답 속도
- **주요 작업:** 데이터 삽입(Insert), 수정(Update), 삭제(Delete)
- **특징:**
  - 정규화: 데이터 중복을 최소화하여 무결성 보장
  - 속도: 밀리초(ms) 단위의 빠른 응답이 필수
  - 예시: ATM 인출, 쇼핑몰 주문, 호텔 예약 시스템

---

## 2. OLAP (Online Analytical Processing)
> "비즈니스 의사결정을 돕는 분석 시스템"

OLTP 등 여러 소스에서 수집된 데이터를 통합하여 복잡한 분석을 수행한다.

- **목적:** 대량 데이터 분석, 트렌드 파악, 비즈니스 인텔리전스(BI)
- **주요 작업:** 복잡한 조회(Select), 집계(Aggregation)
- **특징:**
  - 다차원 분석: OLAP 큐브를 통해 다양한 각도(지역, 시간, 제품 등)에서 데이터 탐색
  - 반정규화: 조회 성능을 높이기 위해 스타 스키마(Star Schema) 등을 사용
  - 예시: 전년 대비 매출 분석, 판매 예측 리포트

---

## 3. 핵심 차이점 요약

| 구분 | OLTP | OLAP |
|------|------|------|
| 초점 | 운영 효율성 및 트랜잭션 | 데이터 분석 및 의사결정 |
| 데이터 소스 | 실시간 비즈니스 데이터 | 다양한 OLTP 데이터베이스의 통합 |
| 데이터 특징 | 최신 데이터 (Snapshot) | 대량의 과거 데이터 포함 (Historical) |
| 쿼리 복잡도 | 단순 (Insert/Update 중심) | 복잡 (집계 및 조인 중심) |
| 응답 시간 | 매우 빠름 (밀리초 단위) | 상대적으로 느림 (초~분 단위) |
| 백업 주기 | 빈번함 (무결성 중요) | 상대적으로 적음 (읽기 전용 위주) |

---

## 💡 DE의 관점: 왜 두 시스템을 나누나요?

운영 DB(OLTP)에서 직접 무거운 분석 쿼리를 실행하면 서비스 성능이 저하된다.
따라서 ETL(Extract, Transform, Load) 과정을 통해 OLTP 데이터를 추출하고 가공하여 분석 전용 저장소인 데이터 웨어하우스(OLAP)로 옮기는 과정이 데이터 엔지니어링의 핵심 프로세스가 된다.

> 부트캠프 최종 프로젝트에서 KAMIS API로 수집한 원시 데이터가 OLTP 성격이었고  
> S3에 저장 후 Trino/Athena로 쿼리하고 분석한 과정이 OLAP 역할을 했다.
